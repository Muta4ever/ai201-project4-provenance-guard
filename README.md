# Provenance Guard

A backend service any creative-sharing platform can plug in to estimate whether a piece of
text was written by a human or generated by AI — then surface that estimate to readers
**honestly**, with a plain-language transparency label, a confidence score that admits
uncertainty, and an appeals path for creators who believe they were misclassified.

Built for AI201 Project 4. Stack: Flask · Groq (`llama-3.3-70b-versatile`) · pure-Python
stylometry · Flask-Limiter · SQLite. Full design rationale lives in
[planning.md](planning.md) (written before any code).

## Architecture overview — the path a submission takes

A submission enters `POST /submit` as `{text, creator_id}`. The raw text is fanned out to
**three independent detection signals**. Each returns an AI-likelihood in `[0, 1]`. Those
scores are combined — weighted, then pulled toward uncertainty when signals disagree — into
a single calibrated `confidence` score. That score is mapped to one of three attribution
verdicts (`likely_ai` / `uncertain` / `likely_human`) and a matching **transparency label**.
The full result plus every signal score is written to the **content store** and the
**structured audit log**, then returned to the caller.

An appeal enters `POST /appeal` with the original `content_id`. The system flips the
content's status to `under_review`, writes an `appeal` event to the audit log (carrying the
original scores *and* the creator's reasoning), and returns a confirmation. No automated
re-classification — the appeal enters a human review queue visible through `GET /log`.

```
POST /submit ─► [LLM signal, structural signal, lexical signal] ─► combine ─► confidence
             ─► label mapping ─► content store + audit log ─► JSON response

POST /appeal ─► lookup by content_id ─► status = under_review ─► audit log (appeal event)
GET  /log    ─► recent structured audit entries (submissions + appeals)
```

Full ASCII diagram of both flows is in [planning.md](planning.md#architecture).

### Endpoints

| Method | Path       | Purpose                                             |
|--------|------------|-----------------------------------------------------|
| POST   | `/submit`  | Classify content → attribution, confidence, label   |

## Detection signals — what each measures, why, and what it misses

The pipeline uses **three distinct signals** (two required; the third earns the *ensemble*
stretch). They are genuinely independent: one semantic, one structural, one lexical.

### Signal 1 — LLM classifier (Groq `llama-3.3-70b-versatile`) · weight 0.50
- **Measures:** holistic semantic/stylistic coherence — does the passage *read* like machine
  text (evenly fluent, hedged, generic) or like a specific human voice?
- **Why I chose it:** captures the gestalt no single statistic can. It's the strongest
  single signal, so it carries the most weight.
- **What it misses:** LLM detectors are known to be **biased against non-native English and
  highly formal human writing**, have no ground truth, and can be fooled by paraphrased AI.
  This is precisely the false-positive danger — so it never decides alone, and the prompt
  explicitly tells the model not to penalize formality by itself.

### Signal 2 — Structural stylometry / "burstiness" · weight 0.25
- **Measures:** sentence-length rhythm — coefficient of variation of sentence length plus
  how uniform sentence lengths are around the mean.
- **Why I chose it:** human writing is *bursty* (mixes long and short sentences); AI writing
  is more uniform. This is a purely structural property the LLM signal doesn't quantify.
- **What it misses:** meaningless on **short texts** and misfires on intentionally uniform
  human forms — **poetry, lyrics, lists, aphorisms**. Reported `reliable=false` below ~4
  sentences / 25 words, and its weight is then redistributed to the LLM signal.

### Signal 3 — Lexical / register · weight 0.25
- **Measures:** vocabulary register — density of informal markers (contractions, filler,
  slang, lowercase "i") vs. AI-favored connectives ("furthermore", "it is important to
  note", "delve", "leverage", "paradigm").
- **Why I chose it:** casual human writing carries informal markers; AI defaults to a
  polished, connective-heavy register. Independent of both rhythm and semantics.
- **What it misses:** **formal human writing** (academic, legal, financial) has no informal
  markers and uses connectives, so it scores AI-like here. Given low weight and cross-checked
  by the LLM for exactly this reason.

## Confidence scoring — how signals combine, and how I validated it

`confidence` is the combined estimate of **P(text is AI)**, in `[0, 1]`:

- `0.50` = maximum uncertainty (a coin flip — the system refuses to make a call)
- toward `1.0` = increasingly confident the text is **AI**
- toward `0.0` = increasingly confident the text is **human**

So `0.51` sits in the uncertain band and produces a *materially different* label than
`0.95` (high-confidence AI). It is not a binary flip at 0.5.

**Combining formula** (see [scoring.py](scoring.py)):

```
raw        = 0.50·llm + 0.25·structural + 0.25·lexical
d          = |llm − stylometry|                       # semantic vs. statistical disagreement
confidence = raw − (raw − 0.5)·d·0.5                  # pull toward 0.5 when signals disagree
```

If a statistical signal is unreliable (short text), its weight is redistributed to the LLM.
If both the LLM and structural signals are unreliable, confidence is forced to `0.50`.

**Asymmetric thresholds — deliberately reluctant to accuse a human:**

| Confidence `c`     | Attribution     | Label variant           |
|--------------------|-----------------|-------------------------|
| `c ≥ 0.70`         | `likely_ai`     | high-confidence AI      |
| `0.30 < c < 0.70`  | `uncertain`     | uncertain               |
| `c ≤ 0.30`         | `likely_human`  | high-confidence human   |

The AI threshold is high (0.70) and the uncertain band wide, because **a false positive
(calling a human's work AI) is worse than a false negative** on a writing platform. The
disagreement penalty reinforces this: mixed evidence lands in "uncertain," not "AI."

**How I validated it's meaningful.** I ran four deliberately chosen inputs (clear AI, clear
human, formal-human borderline, edited-AI borderline) through the pipeline and confirmed the
scores span all three bands and that the false-positive guard works:

| Input                              | llm | struct | lex | confidence | verdict         |
|------------------------------------|-----|--------|-----|------------|-----------------|
| Clear AI (paradigm/furthermore)    | 0.80| n/a*   | 1.00| **0.815**  | likely_ai       |
| Clear human (ramen review)         | 0.20| 0.16   | 0.00| **0.162**  | likely_human    |
| Formal human (monetary policy)     | 0.70| n/a*   | 0.50| **0.635**  | uncertain       |
| Edited AI (remote work)            | 0.20| n/a*   | 0.24| **0.217**  | likely_human    |

`*` structural marked unreliable (fewer than 4 sentences); weight redistributed to LLM.

The key result is the **formal-human** case: even though the LLM alone rated it 0.70 (AI),
the combined score was held at **0.635 → "uncertain"**, not "likely AI." That is the
false-positive asymmetry doing its job.

## Transparency label — all three variants (verbatim)

The label returned by `/submit` changes with the confidence score. `{pct}` is phrased in the
label's own direction (e.g. `c=0.05` → "95% likely human-written").

**High-confidence AI** (`c ≥ 0.70`):
> **⚠ Likely AI-generated**
> Our analysis suggests this content was most likely produced with the help of a generative AI tool. This is an automated estimate, not a certainty.
> Confidence: {pct}% likely AI-generated. The creator can contest this result.

**High-confidence human** (`c ≤ 0.30`):
> **✓ Likely human-written**
> Our analysis found no strong signals of AI generation — this reads as human-written. This is an automated estimate, not a guarantee of authorship.
> Confidence: {pct}% likely human-written.

**Uncertain** (`0.30 < c < 0.70`):
> **◑ Origin uncertain**
> Our analysis was inconclusive. We can't reliably tell whether this was written by a person or generated by AI, so we are not making a determination. Treat the origin as unknown.
> The creator can add context.

Design note: the uncertain label deliberately shows **no percentage** and makes no
accusation — surfacing a number for a coin-flip result would imply false precision. The AI
label always names the appeal path, since that's the costliest verdict for a creator.

## Appeals workflow

- **Who:** the creator of a classified piece of content (via its `content_id`).
- **What they provide:** `content_id` + `creator_reasoning` (free-text explanation).
- **What the system does:** looks up the original decision (404 if unknown), sets status to
  `under_review`, and writes an `appeal` audit event carrying the original
  attribution/confidence/signal scores **alongside** the creator's reasoning.
- **What a reviewer sees:** `GET /log` interleaves the original `classified` entry (with all
  signal scores) and the `appeal` entry (with the reasoning) — everything needed for a manual
  decision, in one place.

```bash
curl -s -X POST http://localhost:5001/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself; I am a non-native English speaker and write formally."}'
```
```json
{
  "content_id": "7b222956-15e9-4391-80ea-16bff2e29005",
  "status": "under_review",
  "message": "Appeal received. This content is now under review by a human moderator. The original classification has been recorded alongside your reasoning.",
  "original_attribution": "likely_ai",
  "original_confidence": 0.815
}
```

## Rate limiting

Applied to `POST /submit` via Flask-Limiter (in-memory store), keyed by client IP:

```python
@limiter.limit("10 per minute;100 per day")
```

**Reasoning for these specific numbers.** A real creator submits their own work occasionally
— a handful of pieces in a sitting at most. **10 per minute** comfortably covers a human
revising and resubmitting a few drafts back-to-back, while stopping a script from hammering
the (paid, latency-bound) Groq-backed endpoint. **100 per day** caps sustained abuse — no
individual creator legitimately needs to classify 100 pieces a day, but that ceiling is high
enough not to block a power user, and it bounds daily LLM-API cost/quota exposure. The two
limits work together: the per-minute cap stops bursts, the per-day cap stops slow-drip
flooding that stays under the minute limit.

**Evidence** — 12 rapid requests from a fresh limiter state (`429` after the 10th):
```
req 1  -> 200      req 7  -> 200
req 2  -> 200      req 8  -> 200
req 3  -> 200      req 9  -> 200
req 4  -> 200      req 10 -> 200
req 5  -> 200      req 11 -> 429
req 6  -> 200      req 12 -> 429
```
A `429` returns a structured JSON body (`{"error": "rate_limit_exceeded", ...}`), not a bare
error page.

## Audit log

Every decision and appeal is written to a structured SQLite table (`audit_log`), surfaced as
JSON via `GET /log`. Each entry records timestamp, content ID, creator ID, event type
(`classified` / `appeal`), attribution, combined confidence, **all three individual signal
scores**, status, and (for appeals) the creator's reasoning — plus the full signal `detail`.

## Known limitations

- **Formal / non-native human writing is our worst case.** A well-structured academic or
  financial paragraph has no informal markers (lexical signal → AI-like) and often uniform
  sentence structure (structural signal → AI-like), and the LLM signal is independently
  biased to flag formal prose as AI. All three can push the same direction. Our mitigations
  (LLM weight, disagreement penalty, high 0.70 threshold) keep the *monetary-policy* test
  case at "uncertain" rather than "likely AI" — but a longer, more uniformly formal human
  essay could still cross into "likely AI." This is a false positive, the costliest error on
  a writing platform, which is exactly why the appeals path exists.
- **Short creative forms (poetry, lyrics, aphorisms).** Too few sentences for reliable
  stylometry, and their deliberate uniformity looks AI-like. We mark structural analysis
  `reliable=false` and lean on the LLM, but confidence on very short text is inherently weak.
- **Adversarial evasion.** A creator (or bad actor) who knows the signals can add
  contractions and vary sentence length to move the score — perfect detection is unsolved,
  which is why the system communicates uncertainty rather than pretending to certainty.

## Spec reflection

- **How the spec helped:** writing the three label variants and the exact combining formula
  in `planning.md` *before* coding meant the implementation was a transcription, not a
  discovery. When I wired up `make_label`, the thresholds and text already existed and just
  had to match — no mid-build design decisions leaking in.
- **How the implementation diverged:** the spec's audit-log sample treats `confidence` as
  essentially "P(AI)" and I kept that, but I also added a **disagreement penalty** that
  wasn't in my first draft of the scoring formula. Testing the formal-human case showed a
  plain weighted average let the LLM's 0.70 drag borderline human writing toward "likely AI";
  pulling the score toward 0.5 when semantic and statistical signals disagree was the honest
  fix, so I added it to both the spec and the code.

## AI usage

1. **Scoring formula generation & correction.** I gave an AI tool the detection-signals and
   uncertainty sections of `planning.md` and asked it to generate `combine()`. Its first
   version implemented a plain weighted average and used symmetric 0.5 thresholds — which
   would have made false positives as likely as false negatives. I overrode it: kept the
   weighting but added the disagreement penalty and the asymmetric 0.70/0.30 thresholds from
   my spec, because the plain version contradicted the platform's cost asymmetry.
2. **Flask skeleton + endpoint stubs.** I directed an AI tool to produce the Flask app with
   `/submit`, `/appeal`, and Flask-Limiter wiring from my API contract. It produced a working
   skeleton but returned bare error strings and used `debug`-only defaults; I revised it to
   return structured JSON errors, a custom `429` handler, input validation on both endpoints,
   and a configurable `PORT` (macOS AirPlay squats on 5000).
